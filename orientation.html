The mission

What the platform must do — collectively, autonomously

A static test suite is not the answer. These capabilities imply distinct agents: generating an attack and judging it are different jobs, and doing both in one context is a conflict of interest by design.

Discover

Generate & mutate

Produce novel adversarial inputs and take a partial success — then autonomously spin variants toward the bypass. Multi-turn, not just single prompts.

Evaluate

Judge independently

Decide success / fail / partial with consistent criteria across runs and target versions — in a context the attack engine can't influence.

Prioritize

Direct the hunt

Read coverage gaps, open findings, and cost; point the Red Team at the highest-value under-tested surface; halt when spend accrues without signal.

Regress

Prove fixes hold

Convert a confirmed exploit into a deterministic, versioned case; re-run on every new target version; catch reappearance and cross-category regressions.

Document

Write the report

Turn a confirmed exploit into a report a senior engineer who wasn't there can reproduce and fix — without a human writing it.

Observe

See under pressure

Surface coverage, pass/fail over versions, resilience trend, open findings, and cost — the substrate the Orchestrator reads to decide.

The attack lifecycle

One attack, end to end

Read the loop as a story. Each hand-off is a typed message in /contracts/v1.

1

Orchestrator picks the target surfacedeterministic

Reads coverage counts, open high-severity findings, and budget from Observability; issues a directive for the highest-value under-tested category — with an authorized scope it sets itself.

→ AttackDirective
2

Red Team generates & attacks the live targetAIHTTP

Builds a multi-turn sequence (novel, or a mutation of a partial success) and runs it against the deployed co-pilot using a real launch-bound token. Captures the raw response.

→ hits POST /api/copilot/*
3

Judge adjudicates — independentlydeterministic-first

Different process, no shared context. It recomputes every signal from the raw response with its own oracles, and reads its own ground truth by correlation-id. The Red Team's hints are advisory only.

← AttackResult → Verdict
4

Verdict lands in the Exploit DBstore

Every outcome persists, pinned by a one-sentence predicate_fired and the target_version — so a later fix passes only when the signal is gone, not when behavior merely drifts.

→ regression case
5

Documentation writes it up — Observability closes the loopAI

A confirmed exploit becomes a reproducible report (critical severity waits for a human). Coverage, cost, and resilience-trend flow back to the Orchestrator, which decides what to hunt next.

→ Vuln report ↺ back to Orchestrator
The four agents

Distinct roles, distinct trust levels

Separation is the whole point — especially Red Team ⟂ Judge. Model choice per role is a deliberate call (see decisions).

Orchestrator
Coordination · deterministic
  • ▸Reads system state, directs the hunt
  • ▸Triggers regression on a new target version
  • ✕Never executes attacks itself
Red Team
Offensive · AI (frontier/open)
  • ▸Generates novel + mutated attacks
  • ▸Multi-turn sequences vs the live target
  • ✕Never evaluates its own attacks
Judge
Adjudication · deterministic-first
  • ▸Recomputes signals with its own oracles
  • ▸Owns its ground truth (by correlation-id)
  • ✕Never shares context with the Red Team
Documentation
Reporting · AI + human gate
  • ▸Confirmed exploit → reproducible report
  • ▸Data-quality checked before write
  • ✕Never files a critical without approval
Trust boundaries

Where the platform stops for a human — and why

Red Team ⟂ Judge — enforced by shape

Separate processes, separate providers. The attacker can't influence the verdict — not even by self-reporting signals. The single most important boundary; a system that generates and evaluates in one context is compromised by design.

Human gate on critical

Low/medium findings auto-file; critical-severity publication needs human approval. A confidently-documented false positive wastes engineering time — so the Documentation boundary must be able to decline.

Remediation is advisory-only

The platform proposes fixes; it never pushes them. A finding that implicates an auth danger-zone is a report, not a patch.

Target-URL allowlist

The platform can only be pointed at the sanctioned target, so it can't be turned against systems it shouldn't attack. It attacks over the guarded HTTP surface only — no core edits, no service accounts.

Why fixes are trustworthy

The regression loop — a test that passes for the right reason

The spec's warning: a test that passes because model behavior changed, not because the vuln was fixed, is worse than no test. The fix is a deterministic predicate + a version axis.

Confirmed exploit→ Pinned by→ Re-run when→ Verdict
attack sequence + oracle predicate_fired target_version changes signal gone = fixed · signal back = regressed · other category flips = cross-category regression

target_version — the resilience axis

The target exposes no build id, so the platform manufactures one (git SHA / deploy metadata). Observability alarms on a sustained-null version rather than letting "is it getting more resilient?" quietly become unanswerable.

How agents talk

Typed contracts, hardened by review

Every hand-off is a versioned JSON Schema in /contracts/v1; both sides are contract-tested. Five schemas, one canonical six-value category taxonomy.

① Orchestrator → Red Team

AttackDirective

Category, mutation axis, authorized scope, budget.

③ Red Team → Judge

AttackResult

Input sequence + raw response + advisory hints.

⑤ Judge → Documentation

Verdict

Outcome, predicate_fired, severity, regression flag.

Judge independence, in the schema

The Red Team's signals travel as observed_hints — contractually advisory. The Judge recomputes. The boundary survives even a drifting attacker.

Generic oracles, not wired findings

Signals are a list of {oracle_id, fired}. A new attack class adds an oracle, not a contract field — the platform stays target-agnostic.

Five typed errors

target_unreachable · budget_exceeded · judge_timeout · no_findings · regression_detected — each closed, each traceable by correlation-id.
